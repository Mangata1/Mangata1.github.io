Practical Secure Aggregation for Privacy-Preserving Machine Learning

OX00 Abstract

The update of federated learning has the problem of privacy leakage. To solve this problem, Google outline an approach to advancing privacy preserving machine learning by leveraging secure multiparty computation to compute sums of model parameter updates from individual users’ devices in a secure manner.

OX01 Cryptographic Primitives

Secret Sharing

Shamir’s t-out-of-n Secret Sharing

Key Agreement

Diffie-Hellman key agreement

Authenticated Encryption

Enc: input key and message and output ciphertext.

Dec: input key and ciphertext and output message.

Pseudorandom Generator

Signature Scheme

UF-CMA secure signature scheme

Secure Aggregation protocol

Initial Phase

  1. All parties are given a security parameter$ k$, which is used for generate $DH $parameters.

  2. Confirm the number of users$ n$ ,and dropoff line threshold$ t$.

  3. Confirm $X_U$ as the data vector we need to transport,$m $is the size of data vector.

  4. Every user u can get a private key from the trusted third party , and user u give other users v a public key.$ ->(d_u^SK,d_v^PK)$

Round 0 : Advertise Key

  1. User u:
    1. Generate two pairs of key $(C_u^{SK},C_u^{PK}), (S_u^{SK},S_u^{PK}) $, and generate $δ_u$ which used Signature Scheme to sign the $C_u^{SK} $and $S_u^{PK}$ using private key $d_u^{SK}.$
    1. Send $(C_u^{PK}, S_u^{PK}, δ_u) $to the server.
  1. Server:
    1. Check if the number of received information$U_1$ less than $t.$if not abort.
    1. Broadcast to all users other’s information $(v,C_v^{PK}, S_v^{PK}, δ_v) $.

Round 1 : Share Key

  1. User u:
    1. Received the information from server:$ (v,C_v^{PK}, S_v^{PK}, δ_v) $, check number more than t , to prevent server from forging the client , we need verify $(δ_v |C_v^{PK}, S_v^{PK})=1.$
    1. Sample a random element$ b_u$ , to be used as a seed for RPG.
    1. Generate t-out-of-n secret share$〖{(v,S_{u,v}^SK)}〗_{vϲU_1} and 〖{(v,b_{u,v})}〗_{vϲU_1 }.$
    1. For other client v use Authenticated Encryption Enc to compute $ e_(u,v)<-AE.enc(private key,u|v|S_{u,v}^{SK} |b_(u,v)) $.
    1. Send all $e_(u,v) $to the server . Note that$ e_(u,v) $contains the address information of $ u $ and $ v$.

2.Server:

    1. Check if the number of received information $U_2$ less than $t.$ if not abort.
    1. Broadcast all users u with other users $ v : e_{u,v} $.

Round 2 : Masked Input Collection

  1. User u:
    1. Coollect $e_(u,v)$ from the server , check the number of received$ e_(u,v) $if less than $ t $ ,abort.
    1. Compute negotiate private key $s_(u,v)$ used $s_(u,v)$ as seed ,send $s_(u,v)$ to the RPG, and then we get a mask $p_(u,v)$.
    1. Compute the other mask $p_(u,)=RPG(b_u).$
    1. Compute the masked input vector $y_u<-x_u+p_u+∑_{vЄU_2} p_{u,v} (mod R).$
    1. Send $y_u$ to the server.
  1. Server:
    1. Check if the number of received information $U_3$ less than $t$.if not abort.
    1. Send $U_3$ (a set of clients) to every client.

Round 3 : Consistency Check

  1. User u:
    1. Check $U_3 $ ,if $U_3$ is smaller than. $t$ , abort.
    1. Compute $δ_u^′<-sign(d_v^{SK},U_3 )$, send $δ_u^′$ to the server.
  1. Server:
    1. Check if the number of received information $U_4$ less than $t$ .if not abort.
    1. Send $(v,δ_v^′)$ to every client.

Round 4 : Unmasking

  1. User u:
    1. Verify $ δ_v^′ $ is or not equal 1.
    1. Get $(u|v|S_{u,v}^{SK} |b_(u,v)$ through $Dec(e_(u,v) )$.
    1. Verify $u=u and v=v$
    1. For dropoff client $v$, send $S_{v,u}^{SK}$to server ,for usual client v, send $b_{v,u} $ to server
  1. Server:
    1. Check if the number of received information$U_5$ less than $t $ .if not abort.
    1. Compute the result.

Algorithm Summary

Why does adding $ b_u $ prevent server spoofing ?

For the client with a large delay, other clients will send $S_uv$. The server may notify other clients that the client is offline. If there is no bu, the server can use $S_uv$ to crack the information from he client with a large delay . After adding $b_u$, although the server obtains the $S_uv$, it still cannot get the $b_u$, so the server can not get the information.

For round 4, why do we need to verify$ U_4 \subseteq U_3$?

It can ensure that the server is not fake. If $v_1$ is declared dead, then $v_1$ does not exist in $U_3$, and the containment relationship does not hold.
